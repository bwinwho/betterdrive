# BetterDrive — CLAUDE.md

Reference for anyone (human or Claude) picking up work on this repo. It covers the architecture, the conventions this codebase leans on, the full feature set as of the current version, the design language, and the release process. Read this before making changes — the app has no test suite, so consistency with existing patterns is what keeps it from rotting.

## What this is

An Electron file manager for Windows. Six fixed "worlds" (tabs) instead of a folder-tree sidebar: Recent, Favourites, Downloads, Pictures, Documents, Cloud (Google Drive, opt-in). Dark theme, serif titles, card-grid browsing, no build step. Ships as a signed-ish NSIS installer via GitHub Actions.

## Architecture

- **`index.html`** — the entire renderer. One file, one classic `<script>` tag (no `type="module"`, no bundler, no framework). All markup, CSS (`<style>` in `<head>`), and JS live here. There used to be an ES-module-based "Preview Engine" (PDF.js/PhotoSwipe/TUI Image Editor) — it was built, tested, and then **fully reverted** after real-device testing showed it broken; previews now just open externally via the OS default app. Don't reintroduce `<script type="module">` without a real reason — it previously required a custom `app://` protocol handler workaround because `file://` blocks ES module imports, which is exactly the kind of fragility this app avoids by policy.
- **`main.js`** — Electron main process. Owns all filesystem access (`ipcMain.handle('fs:*', ...)`), the Google OAuth PKCE loopback flow (`cloud:*`), the HideOut vault (`hideout:*`), standby/suspend logic, and window management. Shells out to PowerShell for things Node/Electron don't expose natively (drive labels via `Get-CimInstance Win32_LogicalDisk`, zip/unzip via `Compress-Archive`/`Expand-Archive`) rather than pulling in a compiled native addon — deliberate, keeps `electron-builder --win` working without a build toolchain. The window is real OS fullscreen (`fullscreen:true`, not just maximized) and `resizable:false`, which means Windows can leave it sized wrong (dead space where a taskbar used to be) after an external monitor is plugged/unplugged — `realignFullscreen()` listens for `screen`'s `display-added`/`display-removed`/`display-metrics-changed` and toggles fullscreen off/on to force Chromium to recompute geometry against whatever display the window is actually on now, since a plain `setBounds` call doesn't reliably trigger that.
- **`preload.js`** — `contextBridge.exposeInMainWorld`, flat `window.localFS.*` / `window.winControl.*` surfaces. Every new IPC channel gets a matching flat method here, named to match the channel (e.g. `hideout:setPin` → `hideoutSetPin`).
- **`standby.html`** — the "Taking a Power Nap ⚡" suspend screen, swapped in after 45s of the window losing focus (`main.js`'s `SUSPEND_DELAY_MS`) to cut idle RAM. `window.__bdSafeToSuspend` in the renderer vetoes suspension mid-edit (e.g. an open rename/modal).
- **`google-config.js`** — gitignored. Holds `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET` for the Cloud world's OAuth. Generated in CI from repo secrets (`release.yml`) or copied locally from `.env` via a small script. Its absence doesn't crash the app — Cloud just shows "not configured."
- **`package.json`** — `electron-builder` config lives in `build`. `asar: false` (deliberate — historically the one packaging trouble spot for worker-bearing libraries, though the Preview Engine that needed it is now gone; left off since there's no reason to re-add asar without a reason to). `files` lists exactly what ships.
- **`scripts/`** — small standalone Node scripts (vendor-copy tooling from the old Preview Engine era; check before assuming still relevant).

## Core conventions

- **`$(sel)` / `$$(sel)`** = `document.querySelector`/`querySelectorAll` shorthand, defined near the top of the script.
- **`S`** — the single global state object. Anything that needs to survive a reload is also mirrored into `localStorage` via **`LS`** (`LS.get(key, default)` / `LS.set(key, val)` / `LS.del(key)`), always namespaced with a `bd_` prefix (e.g. `bd_favourites`, `bd_colorTags`). Don't invent a second state container — add a field to `S` and, if it needs to persist, seed it from `LS.get` and write through `LS.set` on change.
- **Local vs. Drive items share one shape — but not every field.** Every file/folder object looks like `{id, name, mimeType, size, modifiedTime, parents, _local}` regardless of source. For local files `id` is the real filesystem path and `_local:true`; for Drive files `id` is the Drive file id and `_local` is absent/false. `isLocalId(id)` (checks for a path separator) is the switch used everywhere that needs to branch behavior (open, rename, trash, move — see `renameItem`/`trashItem`/`moveItem` for the pattern). `localToItem(e, parentPath)` converts a raw `{path,name,isDir,size,mtimeMs,mime}` filesystem entry (the shape `window.localFS.list`/`hideoutList`/etc. return) into this common item shape — reuse it rather than hand-rolling the mapping again. Some fields are Drive-only and simply don't exist on a local item — `appProperties` in particular (used for link cards' URL, `f.appProperties.bdlink`) is never set by `localToItem`, since local link files store their URL in the file's own content instead (read via `window.localFS.readFile`, see `openItem`'s `k==='link'` branch). Any new code that touches `.appProperties` on a `kind(f)==='link'` item **must** check `f._local` first — reading it unconditionally shipped once already and crashed `fileCard()` (and with it, the entire folder's render) the moment a local `.bdlink` file existed anywhere in that listing.
- **Worlds ("zones") are exactly six, fixed, rebuilt every boot**, not user-creatable: Recent, Favourites, Downloads, Pictures, Documents, Cloud. `rescanLocalWorlds()` reconciles `S.zones` against the real OS special folders on every launch. Favourites is a virtual singleton (`FAVOURITES_ID` sentinel, no real directory). Recent merges Windows' own Recent Items with anything opened through the app. Cloud is Drive's own root (`driveId:'root'`), opt-in only — there's no automatic Drive folder seeding anymore. All are `protected:true` (can't be renamed/removed via the zone menu).
- **Card rendering is centralized.** `folderCard(f,i,num)` / `fileCard(f,i,num)` build the actual grid-item DOM (thumbnail/native-icon/typeglyph, title, meta line, idx badge, color-tag border) and internally call `wireCard(el, f, openFn)` to wire the common interactions: select checkbox, favourite star, lock-to-HideOut button (shown when `f._local`), "more" (⋯) context menu, inline rename-on-title-click, right-click menu, click-to-open (or click-to-select if a selection is active). **Any new view that wants the same visual card grid should call these two functions directly and only override what's genuinely different afterward** — see `vaultCardActions()` in the HideOut section for the established pattern (strip the buttons that don't apply in that context, add the ones that do, reassign `.onclick`/`.oncontextmenu`). Don't reimplement the card markup by hand; visual drift is exactly the kind of bug this pattern exists to prevent.
- **Full-screen overlay pattern.** `#pcbrowse` (import from anywhere on the PC) and `#hideout` (the vault) are both `position:fixed;inset:0` overlays with a `-scrim` sibling that fades in behind them, toggled via a `.show` class, closed via Escape (wired into the single global keydown handler's Escape chain — **`modalOpen` must be checked first** in that chain since `#modal` sits at the highest z-index and can be opened from within either overlay). If you add another full-screen surface, follow this exact shape rather than inventing a new one.
- **Context menus** go through `popMenu(x, y, buttons)` — a generic floating `.menu` builder. `buttons` is an array of `{label, fn, danger?}` / `'hr'` / `{swatches, current, onPick}` (the color-tag row). It's `position:fixed`, so never leave `right` unset on it if you're also setting `left` — a shared `.menu` CSS class carries `right:0` for the (unrelated, `position:absolute`) sort/account dropdowns, and it will stretch the popup full-width if not explicitly cleared. This exact bug happened once already.
- **CSS specificity trap to remember**: `.menu button` (class + tag = higher specificity than a single class) will silently win over any single-class button style nested inside a menu (e.g. `.swatch`) unless that style is written with at least two classes (`.menu-swatches .swatch`) or explicit resets. This exact bug shipped once (color swatches rendered as full-width bars instead of small dots) before being caught — when styling something that's a `<button>` inside `.menu`, either scope the selector with an extra class or explicitly redeclare every property `.menu button` sets.
- **Never put `transform` (or `filter`) on `#app` itself.** `#app` wraps `#zonebar`/`#bottom-scrim`/`#selectbar`/`#clipbar`, all `position:fixed`. Any non-`none` transform or filter on an ancestor — including a finished animation's fill-mode holding `scale(1)` forever — makes that ancestor a containing block for `position:fixed` descendants, un-pinning them from the viewport in favor of tracking the ancestor's own box instead. This exact bug shipped once already: an entrance animation on `#app` used `transform:scale()`, and the zonebar ended up thousands of pixels down the page on any world with enough content to make `#app` taller than one screen, invisible until scrolled all the way down. `#app`'s own entrance animation must stay opacity-only.
- **No automated test suite.** The established verification method, used for every nontrivial renderer change across this project's whole history, is a throwaway headless-Chromium (Playwright) script: mock `window.localFS` / `window.winControl` via `page.addInitScript(...)`, `page.goto('file://.../index.html')`, drive the DOM (`page.click`, `page.fill`, `page.keyboard.press`, `page.evaluate(...)` dispatching real events), and assert on real computed DOM state (`getBoundingClientRect()`, `getComputedStyle()`, class lists) — not just "did the function get called." This caught real bugs that reading the code alone missed (a stale-search race condition that only reproduced with a specifically-timed wait; the CSS specificity bugs above). Write one before shipping, don't skip it because the change "looks obviously correct."
- **Honesty over polish when describing security/robustness.** HideOut is documented (in `main.js`, in the README, and to the user) as Windows' Hidden-file-attribute + an app-level PIN — **not** disk encryption — because it isn't one, and overselling it would be actively misleading. Apply the same standard to anything else that sounds more bulletproof than it is.

## Feature inventory (current)

**Worlds & browsing**
- Six fixed worlds: Recent, Favourites, Downloads, Pictures, Documents, Cloud (Google Drive, opt-in via "+ Connect").
- Recent merges real Windows Recent Items with in-app-opened files, deduped, capped, newest first.
- Favourites: star any file/folder from any world; shows up in the virtual Favourites world with a gold border; local favourites re-stat for freshness, Drive favourites use cached metadata.
- Folder/file cards: thumbnails for local & Drive images, native Windows shell icons for other local files (via `app.getFileIcon`, lazy-loaded on scroll via `IntersectionObserver`), per-kind tinting as a baseline (`KIND_TINT`).
- Full-bleed "image-cover" cards for images — the photo fills the whole card, title/meta sit on a bottom gradient scrim at ~30% opacity, full opacity on hover (so the filename doesn't fight the photo).
- Color tags: right-click any file/folder → 6-swatch picker (blue/red/purple/pink/green + clear) → tints the card's border and cursor-following hover-glow to match. Purely a client-side label (`localStorage`, keyed by item id) — not written to the file itself, layered on top of (and visually wins over) the per-kind tint and the favourite-gold border.
- Command palette (Ctrl+K / Ctrl+P) — jump to a world, run an action, or find a file.
- "Browse PC…" — a separate full-screen-style dialog to import any file/folder from anywhere on disk into the current world (copy or move).
- Search (debounced, per-world local search + a "This PC" system-wide mode); Escape / the on-screen ESC button fully cancels an in-flight search, not just the visible state.
- Full keyboard navigation: digits pick a world, same digit again drops into card-browse mode where digits open the numbered card; arrow keys move focus; `R` renames, `N` new folder, `/` or `S` focuses search, `?` opens help, `Backspace`/Escape step back out.

**File operations**
- Create folder, rename (inline on the title text specifically, not the whole card), move to trash (Recycle Bin for local, Drive trash for Drive), move to a different world.
- Clipboard flow: Copy/Cut swaps the selection bar for a Paste bar so you can navigate anywhere and paste, instead of being stuck; Cut disables re-selecting more items until you paste or cancel.
- Bulk selection bar: copy / cut / move / favourite / zip / trash across a multi-selection.
- `.exe` launches directly on open; `.zip` auto-extracts (via PowerShell `Expand-Archive`/`Compress-Archive`) and drops you straight into the extracted folder; zip-selected creates one from a multi-selection.
- Drag-and-drop with spring-eased lift/drop animations; OS drag-and-drop upload works too.

**HideOut (secret vault)**
- A "HideOut" button next to search: first use sets a PIN (entered twice to confirm), every subsequent use verifies it — no session-level "stay unlocked," it re-asks every time you open the vault. PIN is stored via `safeStorage` (the same OS-level encryption already used for the Cloud OAuth tokens), not plaintext.
- A lock button (next to the favourite star, on every local file/folder card, replacing what used to be a Recent-only "Move" shortcut) moves the item into the vault folder and applies the Windows Hidden attribute (`attrib +h`) to it — so it also disappears from a normal Explorer listing, not just from BetterDrive.
- The vault itself opens as a **full-screen world**, not a small dialog — same folder/file card grid, thumbnails, native icons, and hover-glow as every other world (see `vaultCardActions()` — it reuses `folderCard`/`fileCard` directly and only swaps the fav/lock/select/more buttons for Restore/Trash, since those don't apply to an already-hidden item). Click a card to open it directly (not through the main app's folder-navigation stack, so its real path never leaks into the visible breadcrumb); right-click for Open/Restore/Trash.
- Honest scope, stated everywhere this feature is mentioned: this is Windows' Hidden attribute + an app-level PIN gate, **not disk encryption**. Anyone with "show hidden files" on, or raw disk access, can still see the files.

**Cloud (Google Drive)**
- OAuth via a main-process PKCE loopback flow (no embedded browser sign-in, no webview) — opens the system browser, listens on a local port for the redirect.
- Full read/write against My Drive; Cloud is opt-in, never auto-seeded.
- Favouriting a file also quietly uploads it to a `BetterDrive/Favourites` Drive folder in the background (best-effort, doesn't block the UI).
- Disconnect available from the account menu.

**Chrome & feel**
- Custom titlebar (Electron-only), fullscreen non-resizable window, dark-first paint (no white flash on launch — see Design language below for why that mattered).
- Standby mode: 45 seconds of the window losing focus swaps in `standby.html` ("Taking a Power Nap ⚡") to cut idle RAM; instant resume, no loading-screen theater. Vetoed mid-edit via `window.__bdSafeToSuspend`.
- Toast notifications for background actions (uploads, moves, errors).
- PWA fallback exists for non-Electron/browser use, but most local-filesystem features are gated to Electron (`IS_ELECTRON`/`window.localFS` presence) and simply don't render outside it.

## Design language

- **Palette**: near-black background (`--bg:#0A0A09`), warm off-white text (`--cream:#EAE4D6`), a muted warm-grey secondary (`--dim:#8E887B`), and translucent cream for hairline borders (`--faint:rgba(234,228,214,.07)`, `--faint-2:rgba(234,228,214,.13)`). Accent colors are used sparingly and each means something specific: gold (`#e8c766`) = favourited, rust-red (`--danger:#C96F5E`) = destructive, purple (`--vault:#7a4aa5`) = HideOut. Per-file-kind tint colors (`KIND_TINT`) and the 5 user-assignable color-tag colors (`COLOR_TAGS`: blue `#3a6ea5`, red `#a5433a`, purple `#7a4aa5`, pink `#a54a86`, green `#3a9463`) are deliberately muted/dark, not saturated — nothing should shout.
- **Type pairing**: serif (`'Cormorant Garamond'`) for anything that's a *name* — folder/file titles, modal headings, world names — sized large (22–42px) for presence; monospace (`'DM Mono'`) in small, uppercase, letter-spaced (`.15–.25em`) caps for anything that's a *label* — metadata, timestamps, buttons, badges, the breadcrumb. Body/paragraph text uses `'DM Sans'`. This split (serif for identity, mono-caps for chrome) is the single most identifying visual trait of the app — keep it when adding anything new.
- **Motion**: one shared easing curve for almost everything (`--ease: cubic-bezier(.22,.61,.36,1)`, a gentle ease-out), a springier one (`--spring: cubic-bezier(.34,1.56,.64,1)`) reserved for drag-and-drop lift/drop. Card entrances stagger via a `--i` custom property multiplied into an `animation-delay`. Hover states are always transitions, never instant. Cards get a cursor-following radial "spotlight" glow (`--mx`/`--my` custom properties set from `pointermove`, consumed by a `::after` radial-gradient) — tinted to match a card's color tag if it has one, otherwise a neutral cream glow.
- **Layout**: no sidebar. A top search bar, a context/breadcrumb row, a `#stage` content area, and a floating pill-shaped world-switcher bar docked at the bottom-center (`#zonebar`) — a deliberate inversion of the conventional file-manager chrome. Folders and files render in separate labeled sections (`.section-label` — small caps with a trailing hairline rule), each as a responsive `auto-fill` CSS grid (`.grid-folders`: min 420px columns, wider/shorter cards; `.grid-files`: min 290px columns, taller cards) rather than a fixed column count, so it reflows naturally at any window size.
- **Card anatomy**: a folder/file icon or thumbnail, a title, a one-line meta row (size · date · extra badges), an index-number badge (top-left, keyboard-shortcut hint, only 1–9), a select checkbox and favourite star (top-right, both mostly-transparent until hover), and a "more" (⋯) button (bottom-right, appears on hover). Any card variant (HideOut's included) should keep this same visual skeleton and only substitute which action buttons appear.
- **First paint matters**: an inline `<style>html,body{background:#0A0A09}</style>` sits before any other `<head>` resource specifically so the window never flashes Chromium's default white background while slower resources (Google Fonts) are still loading. Don't reorder the `<head>` without preserving this.
- **Full-screen over floating dialogs, where it makes sense**: secondary "browse a bunch of files" surfaces (PC import, HideOut) are full-viewport, not small centered modals — small modals (`#modal`) are reserved for short single-purpose interactions (confirm/rename/prompt/PIN entry), not anything with a scrollable list of its own.

## Release process

1. Bump `version` in `package.json`.
2. Commit, push to `main`.
3. `.github/workflows/release.yml` builds on `windows-latest` via `npx electron-builder --win --publish always`, publishing straight to GitHub Releases (not draft). It triggers on a pushed `vX.Y.Z` tag **or** manually via `workflow_dispatch` — the build always reads its version from `package.json`, not from the tag, so `workflow_dispatch` alone is a fully valid way to ship if tag-pushing isn't available in your environment.
4. No CI test suite gates this — verify locally (headless-Chromium script per the convention above) before pushing.
